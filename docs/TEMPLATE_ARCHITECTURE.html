Why raw ASGI? BaseHTTPMiddleware wraps responses in a new task, breaking contextvars propagation through FastMCP's streaming transport.

Two Authentication Schemes

① API Key

Simple header-based auth. No Azure setup required.

# Request
POST /mcp
x-api-key: your-secret-key

# .env
API_KEYS=key-one,key-two

Use case

  • Server-to-server integration
  • Local development without Azure
  • CI/CD pipelines and scripts
Auth context: auth_type = "api_key"
No user identity — user_oid is None

② OAuth 2.0 Bearer Token

Azure Entra JWT — full user identity in every request.

# Request
POST /mcp
Authorization: Bearer eyJ0eXAiOiJKV1Q...

# Token claims (decoded)
{
  "aud": "api://<AZURE_CLIENT_ID>",
  "oid": "b6167420-...",  ← user object ID
  "scp": "access_as_user",
  "name": "Xindong Yang"
}

Use case

  • Claude.ai / Claude Desktop users
  • Copilot Studio connectors
  • Any OAuth 2.1 client
Auth context: auth_type = "bearer"
user_oid · user_name · user_upn all populated

Authentication Middleware — Decision Flow

class AuthenticationMiddleware:

  async def __call__(scope, receive, send):

    # 1. Exempt paths — bypass auth entirely
    if path in {"/", "/health"} or
       path.startswith("/.well-known/", "/docs",
                       "/register", "/authorize", "/token"):
        await next_app()
        return

    # 2. Try Bearer token first
    if header.startswith("Bearer "):
        user_oid, name, upn = await jwt.authenticate(token)
        set_auth(AuthContext(auth_type="bearer", ...))
        await next_app()          ← ✅ pass through

    # 3. Fallback: x-api-key
    elif api_key:
        ensure_valid_api_key(api_key)
        set_auth(AuthContext(auth_type="api_key"))
        await next_app()          ← ✅ pass through

    # 4. No credentials → 401
    else:
        return 401 + WWW-Authenticate: Bearer
                   resource_metadata=".../.well-known/..."

AuthContext — propagated via contextvars

@dataclass
class AuthContext:
    auth_type: str   # "bearer"|"api_key"
    user_oid:  str | None
    user_name: str | None
    user_upn:  str | None

from src.auth_context import get_auth

@mcp.tool()

OAuth 2.0 — Auto-Discovery Chain

Clients discover all OAuth endpoints automatically. No manual configuration needed.

Step 1 — Client hits /mcp without a token
─────────────────────────────────────────────
POST /mcp  (no token)
  ← 401 WWW-Authenticate: Bearer
           resource_metadata="https://host/.well-known/
                               oauth-protected-resource"

Step 2 — Client fetches RFC 9728 metadata
─────────────────────────────────────────────
GET /.well-known/oauth-protected-resource
  ← {
      "resource":             "https://host/mcp",
      "authorization_servers": ["https://host"]  ← our proxy, NOT Entra
    }

Step 3 — Client fetches RFC 8414 AS metadata
─────────────────────────────────────────────
GET /.well-known/oauth-authorization-server
  ← {
      "issuer":                 "https://host",     ← MUST match fetch origin
      "authorization_endpoint": "https://host/authorize",
      "token_endpoint":         "https://host/token",
      "registration_endpoint":  "https://host/register"
    }

Step 4 — Client proceeds with OAuth flow

Why issuer = our server, not Entra?

If issuer pointed to Entra, Claude would detect the mismatch between our AS metadata and Entra's, reject our proxy, and call Entra directly — sending a resource parameter that Entra v2 rejects (AADSTS9010010).

Why is api://xxx/access_as_user hidden?

If this Azure-specific scope appeared in scopes_supported, clients would detect the Entra pattern and bypass our proxy. The proxy injects it silently at /authorize.

OAuth 2.0 Proxy — Azure Entra Bridge

Azure Entra doesn't support RFC 7591 DCR. The proxy bridges this gap.

① POST /register  (RFC 7591 DCR)
   Client → {redirect_uris, client_name, grant_types}
   Server → {client_id: "dynamic-xxxx", token_endpoint_auth_method: "none"}
            └ stored in _registered_clients{} (in-memory)

② GET /authorize
   Client → ?client_id=dynamic-xxxx&scope=openid+profile
             &code_challenge=...&redirect_uri=...

   Proxy  → swap client_id → AZURE_CLIENT_ID
          → inject scope: api://xxx/access_as_user + offline_access
          → drop  resource param  (Entra v2 rejects it)
          → 302 Entra /oauth2/v2.0/authorize

③ User logs in at Entra → code returned to redirect_uri

④ POST /token
   Client → {grant_type=authorization_code, code=xxx,
              code_verifier=..., client_id=dynamic-xxxx}

   Proxy  → swap client_id     → AZURE_CLIENT_ID
          → swap client_secret  → AZURE_CLIENT_SECRET
          → drop resource param
          → forward to Entra /oauth2/v2.0/token

⑤ Entra → {access_token, id_token, refresh_token}
   access_token.aud = "api://AZURE_CLIENT_ID"  ✓

What clients see

  • A generic OAuth 2.1 server
  • Standard DCR endpoint
  • No Azure-specific scopes
  • No Entra URLs visible

What Entra sees

  • Real AZURE_CLIENT_ID
  • Real AZURE_CLIENT_SECRET
  • Correct api:// scope
  • No resource parameter

scope is optional at /authorize

Some clients (Copilot Studio) omit scope. The proxy always injects the required api:// scope regardless, starting from an empty set if needed.

JWT Token Validation (Resource Server Role)

async def authenticate(token: str):

    # 1. Extract kid from token header
    kid = jwt.get_unverified_header(token)["kid"]

    # 2. Fetch Entra JWKS  (cached 1 hour)
    jwk = fetch_from_entra_jwks(kid)
    public_key = rsa_pem_from_jwk(jwk)

    # 3. Verify signature + audience + expiry
    payload = jwt.decode(
        token, public_key,
        algorithms=["RS256"],
        audience=valid_audiences,    ← see table →
        options={"verify_exp": True}
    )

    # 4. Manual issuer check  (v1 + v2 tokens)
    if payload["iss"] not in valid_issuers:
        raise InvalidIssuerError

    # 5. Require oid claim
    user_oid = payload["oid"]           ← user identity

    return user_oid, payload.get("name"), payload.get("upn")

Stream Concurrency Control

Each MCP tool call holds a streaming HTTP connection open. The StreamConcurrencyController prevents resource exhaustion.

Incoming request
        │
        ▼
Semaphore.acquire(timeout=QUEUE_WAIT_TIMEOUT)
        │
   ┌────┴─────────────────────────────────┐
   │ Timeout?  →  429 Too Many Requests  │
   │              Retry-After: {timeout}  │
   └────┬─────────────────────────────────┘
        │ acquired
        ▼
asyncio.create_task(app_runner)
        │
        ├── idle_watchdog: cancel if no bytes sent for IDLE_STREAM_TIMEOUT
        └── hard_watchdog: cancel after HARD_STREAM_TIMEOUT regardless
              │
        CancelledError → 503 Service Unavailable
              │
        finally: Semaphore.release()

Configuration

VariableDefaultEffect
MAX_CONCURRENT_STREAMS20Semaphore size
QUEUE_WAIT_TIMEOUT5 sWait before 429
HARD_STREAM_TIMEOUT300 sMax request life
IDLE_STREAM_TIMEOUT60 sMax silence

Health endpoint exposes live metrics

GET /health  →
{
  "mcp": {
    "active_streams": 3,
    "max_streams": 20
  }
}
MCP streams can be long-lived (seconds to minutes). Without limits, one busy burst could exhaust memory or downstream DB connections.

Getting Started

① Configure .env

# Required
BASE_URL=https://your-host.com
API_KEYS=your-secret-key

# Azure Entra (OAuth2)
AZURE_TENANT_ID=<tenant-id>
AZURE_CLIENT_ID=<app-registration-id>
AZURE_CLIENT_SECRET=<client-secret>

# Optional tuning
MAX_CONCURRENT_STREAMS=20

② Add your tools in src/server.py

@mcp.tool()
async def my_tool(param: str) -> str:
    """Tool description shown to the LLM."""
    auth = get_auth()
    # auth.auth_type, auth.user_oid, etc.
    return "result"
